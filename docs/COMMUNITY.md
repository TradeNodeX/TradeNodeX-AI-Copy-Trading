# TradeNodeX Community Guide

This project is intended to be practical, security-conscious, and operator-focused.

## Recommended GitHub structure

### Issues

Use Issues for:

- Reproducible bugs
- Clearly scoped feature requests
- Documentation defects
- Compatibility reports with clear environment details

Do not use Issues for:

- Private credentials
- Production account screenshots
- Security vulnerabilities
- Open-ended usage consulting

### Discussions

Use Discussions for:

- General questions
- Deployment notes
- Integration ideas
- Exchange-specific observations
- Community examples
- Roadmap feedback

Recommended Discussion categories:

- Announcements
- Q&A
- Ideas
- Show and tell
- Deployment notes
- Exchange compatibility

### Pull requests

Use pull requests for focused improvements. Each PR should explain what changed, why it changed, and how it was validated.

## Community tone

- Be precise.
- Be respectful.
- Avoid hype and unverifiable claims.
- Prefer operational facts, logs without secrets, and reproducible steps.
- Treat security and credentials seriously.

## Maintainer response model

Recommended triage labels:

- `bug`
- `documentation`
- `enhancement`
- `security`
- `exchange-adapter`
- `deployment`
- `frontend`
- `backend`
- `good-first-issue`
- `help-wanted`
- `question`

## Public communication principle

TradeNodeX should be presented as self-hosted operational infrastructure. Public messaging should avoid return claims, performance guarantees, or language that implies managed financial services.
